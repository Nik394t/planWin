# Поэтапный промт / ТЗ на реализацию веб-версии Daily

## 1. Роль исполнителя

Ты — senior frontend/PWA инженер. Нужно создать веб-версию приложения `Daily` в отдельной папке проекта, не трогая мобильное приложение `Daily` на Flutter.

## 2. Цель

Сделать отдельную веб/PWA-версию `Daily`, визуально и функционально максимально близкую к текущему мобильному приложению:
- app-like интерфейс,
- адаптация под любой экран,
- сохранение данных на устройстве даже после закрытия вкладки,
- стабильная работа всех сценариев,
- поддержка уведомлений,
- аккуратный брендинг с логотипом и ощущением реального установленного приложения.

## 3. Ограничения

- Не изменять папку `/Users/pro/Desktop/ЗАВЕРШЕННЫЕ ПРОЕКТЫ/plan_win_bot/Daily`.
- Новый проект создать отдельно в `/Users/pro/Desktop/ЗАВЕРШЕННЫЕ ПРОЕКТЫ/plan_win_bot/Daily_web`.
- Интерфейс не должен быть шаблонным; нужен выразительный темно-золотой стиль в духе текущего Daily.
- Все критичные операции должны быть безопасны при быстрых кликах, закрытии модалок, смене страниц, перезагрузке вкладки.

## 4. Рекомендуемый стек

Использовать:
- `React + TypeScript`
- `Vite`
- `Zustand` для UI/state orchestration
- `Dexie` поверх `IndexedDB` для локального хранения
- `React Router`
- `date-fns` или `dayjs` для периодов и timezone-конвертации
- `Service Worker`
- `Web App Manifest`
- `Web Push` с backend scheduler для надежных фоновых уведомлений

Если выбран другой стек, он должен сохранять те же гарантии: PWA, IndexedDB, service worker, миграции и тестируемость.

## 5. Почему именно PWA, а не просто сайт

Нужна установка на телефон как приложения:
- иконка на домашнем экране,
- splash screen,
- полноэкранный режим,
- локальные данные на устройстве,
- service worker,
- offline shell,
- нативное ощущение на Android и максимально близкое на iPhone.

## 6. Ключевое техническое требование по данным

Данные должны сохраняться на устройстве и оставаться после:
- закрытия вкладки,
- закрытия браузера,
- перезагрузки телефона,
- обновления самой веб-версии.

Для этого:
- хранить сущности в `IndexedDB`,
- не хранить бизнес-данные только в `localStorage`,
- ввести `schemaVersion` и миграции,
- при обновлении приложения делать миграции без удаления старых данных,
- service worker не должен чистить пользовательские данные.

## 7. Ключевое техническое требование по уведомлениям

Нужно реализовать 3 уровня уведомлений:

1. Глобальные уведомления Daily:
- утреннее уведомление,
- вечернее уведомление,
- оба настраиваемые,
- timezone управляется из настроек.

2. Напоминания по задачам:
- опциональны,
- по умолчанию выключены,
- включаются тумблером внутри формы задачи,
- для одной задачи можно создать несколько напоминаний.

3. Правила завершения напоминаний:
- если задача выполнена, все её будущие уведомления отменяются,
- если период завершен или прошел, уведомления старого периода больше не приходят,
- если задача с замочком, напоминания продолжаются в следующем релевантном периоде,
- если задача удалена, её уведомления удаляются.

### Важное ограничение веб-платформы

Нельзя обещать поведение 1-в-1 с Android native без backend push-инфраструктуры.

Поэтому архитектура уведомлений должна быть такой:
- локально хранить расписание напоминаний,
- на сервере хранить подписку `PushSubscription`,
- сервер планирует отправку web push по UTC,
- service worker показывает уведомление на устройстве,
- клиент после каждого изменения задачи/настроек синхронизирует расписание на backend.

Без сервера веб не даст надежных фоновых уведомлений после полного закрытия приложения на всех устройствах.

## 10. Правила UX и стабильности

Обязательные правила:
- не вызывать навигацию внутри render/build без эффекта,
- не выполнять `setState`/state update после unmount,
- после каждого `await` проверять актуальность mounted/route state,
- блокировать повторное сохранение кнопкой `isSaving`,
- все формы должны возвращать результат вверх, а не менять внешний список напрямую,
- любой `double-tap` по `Сохранить` не должен создавать дубли,
- закрытие модалки во время async не должно вызывать crash,
- возврат назад во время async не должен ломать state.

## 13. Логика задач и периодов

### 13.1 Создание задачи
- Форма открывается как modal sheet/dialog/page.
- Поля: название, описание, тумблер напоминаний, список напоминаний, замочек.
- Для дневного плана при добавлении напоминания выбрать только время.
- Для недельного/месячного/годового плана выбрать дату внутри периода, потом время.
- Прошедшие дата/время недоступны.
- После подтверждения форма возвращает результат вверх.

### 13.2 Завершение задачи
- При `isDone = true` удалить все будущие уведомления этой задачи.
- Зафиксировать `doneAt`.
- Записать событие в историю.

### 13.3 Замочек
- Если задача закреплена, она переносится как recurring pattern в следующий соответствующий период.
- Если у неё включены напоминания, они тоже переносятся.
- Если замочек снят, будущие автогенерации прекращаются.

### 13.4 Завершение периода
- Когда день/неделя/месяц/год завершились, старый период становится только для чтения.
- Для старого периода записывать итог в историю.
- Все уведомления прошедшего периода отменять.
- При наступлении нового дня экран `Планы` автоматически переключать на новый текущий день.

## 16. Проверка уведомлений

Нужно реализовать и протестировать:

- сохранение подписки,

Тексты:

- вечер: если до `18:00` или настроенного времени остались невыполненные пункты, прислать сообщение вида: `Уже вечер, а план на сегодня еще не закрыт. Осталось N пунктов. Зайди в Daily и давай завершим его сегодня.`
- задача: `Напоминаю о задаче: <текст задачи>`

- есть тесты на критичные сценарии,
- нет крашей при быстрых действиях пользователя.
